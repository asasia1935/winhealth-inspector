# Architecture Draft

> 이 문서는 초기 설계 방향을 기록한 초안입니다. 실제 첫 번째 진단 기능을 구현하면서 프로젝트 구조와 모델은 조정할 수 있습니다.

## 기본 원칙

- UI와 Windows 시스템 접근 코드를 분리한다.
- 데이터 수집과 상태 판정을 분리한다.
- 진단과 시스템 변경 작업을 분리한다.
- 실제 Windows API 없이 판정 규칙을 테스트할 수 있어야 한다.
- 하나의 진단이 실패해도 전체 진단은 계속 진행되어야 한다.
- 기본 실행에서는 관리자 권한을 요구하지 않는다.
- 검사 실패를 정상 상태로 처리하지 않는다.
- MVP 단계에서 불필요한 플러그인 구조나 과도한 추상화를 도입하지 않는다.

## 예상 프로젝트 구조

```text
WinHealthInspector
├─ WinHealth.App
├─ WinHealth.Core
├─ WinHealth.Infrastructure.Windows
└─ WinHealth.Tests
```

정확한 프로젝트 수와 이름은 구현 시작 전에 확정한다.

## 프로젝트 역할

### WinHealth.App

WPF UI와 사용자 상호작용을 담당한다.

- View
- ViewModel
- Command
- Converter
- Style 및 Resource
- Navigation
- 진단 및 관리 작업 상태 표시

### WinHealth.Core

Windows에 직접 의존하지 않는 핵심 모델과 규칙을 담당한다.

- 진단 인터페이스
- 관리 작업 인터페이스
- 진단 결과 모델
- 관리 작업 결과 모델
- 상태 enum
- Snapshot 모델
- 판정 규칙
- 공통 오류 모델

### WinHealth.Windows

실제 Windows 정보 수집과 Windows 관련 작업을 담당한다.

- 드라이브 정보
- 사용자 임시 파일 탐색 및 정리
- 이벤트 로그
- Windows 보안 상태
- Windows 업데이트 상태
- 시작 프로그램
- 네트워크 상태
- 휴지통
- DNS 캐시 초기화

### WinHealth.Tests

실제 Windows API와 분리된 규칙과 흐름을 검증한다.

- 판정 규칙 테스트
- 진단 실행 흐름 테스트
- ViewModel 테스트
- 관리 작업 미리보기 테스트
- 경로 안전성 테스트
- 취소 및 시간 초과 테스트

## 진단 흐름

```text
Windows Provider
→ Snapshot
→ Diagnostic Rule
→ Diagnostic Result
→ ViewModel
→ View
```

### 데이터 수집

Windows Provider는 운영체제에서 원시 상태를 읽고 Snapshot을 만든다.

이 단계에서는 가능한 한 상태를 해석하지 않는다.

### 판정

Diagnostic Rule은 Snapshot을 기반으로 상태를 판정한다.

- 실제 측정값
- 상태
- 판정 기준
- 예상 영향
- 권장 조치

판정 규칙은 실제 Windows API 없이 테스트할 수 있어야 한다.

### 표시

ViewModel은 결과를 화면 표시 형태로 관리한다.

Windows API를 직접 호출하거나 시스템 상태를 직접 판정하지 않는다.

## 관리 작업 흐름

```text
Diagnostic Result
→ Available Action
→ Preview
→ User Confirmation
→ Execute
→ Action Result
→ Related Diagnostic Recheck
→ Before/After Comparison
```

### 미리보기

관리 작업은 실행 전에 가능한 범위에서 다음 정보를 제공한다.

- 작업 대상
- 예상 파일 수 또는 영향 범위
- 예상 확보 용량
- 위험 수준
- 관리자 권한 필요 여부
- 되돌리기 가능 여부

### 실행

관리 작업은 허용된 범위만 변경한다.

- 사용자가 선택한 작업만 실행
- 취소 가능한 구간 명확화
- 일부 성공 허용
- 실패 항목 기록

### 재검사

작업이 완료되면 관련 진단을 다시 실행하여 전후 결과를 비교한다.

## 핵심 추상화 초안

구체적인 인터페이스와 모델은 첫 수직 기능을 구현하면서 확정한다.

### Diagnostic Check

각 진단은 독립적으로 실행할 수 있어야 한다.

예상 책임:

- 진단 식별 정보 제공
- 취소 토큰 수신
- 진단 결과 반환

```text
각 진단은 예상 가능한 환경 차이와 지원 여부를 결과로 반환한다.

처리되지 않은 예외, 시간 초과 및 공통 실행 실패는
Diagnostic Runner가 포착하여 `Unknown` 결과로 변환한다.

예외를 정상 상태로 변환하거나 조용히 무시하지 않는다.
```

### Maintenance Action

각 관리 작업은 진단과 분리되어야 한다.

예상 책임:

- 작업 식별 정보 제공
- 위험 수준 제공
- 관리자 권한 필요 여부 제공
- 되돌리기 가능 여부 제공
- 미리보기 생성
- 실행 결과 반환

### Diagnostic Runner

전체 진단 실행을 조정한다.

예상 책임:

- 여러 진단 실행
- 진행률 보고
- 취소 처리
- 진단별 시간 초과
- 일부 실패 허용
- 결과 수집 및 정렬
- 중복 실행 방지

## 비동기 및 실패 정책

- UI 스레드를 차단하지 않는다.
- 장시간 작업은 비동기로 실행한다.
- 진단별로 시간 초과를 적용할 수 있어야 한다.
- 하나의 진단 실패가 전체 실행을 중단하지 않는다.
- 예외를 무시하거나 정상으로 변환하지 않는다.
- 실패 원인을 사용자에게 노출할 수 있는 안전한 메시지로 변환한다.
- 원본 예외와 민감한 경로는 기본 UI나 내보내기 보고서에 그대로 노출하지 않는다.

## 권한 방향

MVP에서는 애플리케이션 전체를 관리자 권한으로 실행하지 않는다.

- 일반 권한으로 가능한 진단과 작업을 우선 제공한다.
- 관리자 권한이 필요한 항목은 `RequiresAdministrator` 또는 `Unknown`으로 처리한다.
- 향후 관리자 작업을 지원할 경우 제한된 별도 Helper 프로세스를 검토한다.

## 저장 방향

MVP에서는 마지막 진단 결과의 로컬 저장과
사용자 요청에 따른 JSON 보고서 내보내기를 지원하는 것을 목표로 한다.

핵심 진단 및 관리 작업의 안정성이 우선이며,
일정에 따라 공개 품질 안정화 단계로 이동할 수 있다.

세부 형식은 구현 전에 확정한다.

원칙:

- 외부 서버 전송 없음
- 사용자 이름과 민감한 전체 경로 최소화
- 사용자가 내보내기를 명시적으로 요청한 경우에만 보고서 생성
- 진단 원본 데이터 전체를 무조건 저장하지 않음

## 취소 정책

- 취소는 가능한 안전한 지점에서 반영하는 최선형 취소로 구현한다.
- 이미 완료된 변경 작업은 취소 요청만으로 되돌리지 않는다.
- 되돌릴 수 없는 작업은 실행 전에 이를 사용자에게 알린다.
- 작업 결과에는 취소 전까지 완료된 항목과 미처리 항목을 구분한다.

## 주요 설계 과제

- 하나의 진단에서 여러 발견 항목이 발생하는 경우의 결과 모델
- 진행률 계산 방식
- 진단별 시간 초과 기본값
- 진단 병렬 실행 범위
- 이벤트 로그 그룹화 기준
- 다른 백신 사용 환경에서의 보안 판정
- 사용자 임시 파일의 안전한 판별 기준
- 취소 중 파일 삭제 작업의 일관성
- 휴지통 용량 조회 방식
- DNS 캐시 초기화의 권한 처리
- 마지막 진단 결과의 로컬 저장 형식

## 프로젝트 참조 방향

```text
WinHealth.App
├─ WinHealth.Core
└─ WinHealth.Windows

WinHealth.Windows
└─ WinHealth.Core

WinHealth.Tests
├─ WinHealth.Core
├─ WinHealth.Windows
└─ 필요한 경우 WinHealth.App

WinHealth.Core
└─ 다른 프로젝트를 참조하지 않음

### 원칙
- Core는 WPF와 Windows API를 참조하지 않는다.
- Windows는 Core에 정의된 계약과 모델을 구현한다.
- App은 Core의 모델과 Windows의 구현을 조립한다.
- ViewModel은 Windows API를 직접 호출하지 않는다.
- 프로젝트 간 순환 참조를 허용하지 않는다.
