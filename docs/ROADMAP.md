# Project Roadmap

> 이 문서는 WinHealth Inspector의 중장기 개발 방향을 기록한 초안입니다.
> 각 단계는 이전 단계의 안정성과 검증이 완료된 이후 진행합니다.

## 기본 원칙

- MVP 기능 완성을 배포 자동화보다 우선한다.
- 한 단계가 안정화되기 전에 다음 단계의 범위를 무리하게 추가하지 않는다.
- 모든 배포 결과물은 자동 빌드와 테스트를 통과해야 한다.
- 개발, 베타, 정식 배포 채널을 가능한 범위에서 분리한다.
- 사용자 시스템 정보와 진단 결과를 배포 인프라로 전송하지 않는다.
- 배포 편의성을 이유로 애플리케이션의 최소 권한 원칙을 훼손하지 않는다.

## Phase 0. 제품 정의 및 기술 결정

### 목표

프로젝트의 목적, MVP 범위, 안전 원칙과 초기 기술 방향을 확정한다.

### 작업

- 제품 정의 작성
- MVP 범위 작성
- 안전 및 개인정보 원칙 작성
- 초기 아키텍처 방향 작성
- Codex 작업 규칙 작성
- .NET 버전 및 테스트 프레임워크 결정
- 솔루션과 프로젝트 구조 생성
- 기본 GitHub Actions 워크플로 구성
- Pull Request 및 main 브랜치에서 자동 빌드·테스트

### 완료 기준

- 문서 간 제품 방향과 MVP 범위가 일치한다.
- 구현을 시작하기 위한 핵심 기술 선택이 완료된다.
- 첫 번째 수직 기능을 구현할 수 있는 저장소 구조가 준비된다.

## Phase 1. MVP 진단 기능

### 목표

Windows의 주요 상태를 안전하고 투명하게 진단한다.

### 범위

- 저장 공간 상태
- 최근 프로그램 및 시스템 오류
- Windows 보안 상태
- Windows 업데이트 및 재부팅 상태
- 시작 프로그램 상태
- 네트워크 기본 상태

### 공통 기능

- 전체 및 개별 진단 실행
- 진행률과 현재 실행 항목 표시
- 취소와 시간 초과
- 일부 진단 실패 허용
- 결과 필터 및 상세 근거 표시
- 마지막 진단 결과 로컬 저장
- JSON 보고서 내보내기

### 완료 기준

- 여섯 개 진단 기능이 실제 Windows 환경에서 동작한다.
- 검사 실패와 시스템 문제를 명확히 구분한다.
- 진단 규칙의 주요 경계값에 대한 단위 테스트가 존재한다.
- 진단 실행 중 UI가 차단되지 않는다.

## Phase 2. MVP 관리 작업

### 목표

진단 결과와 연결된 안전한 관리 작업을 제공한다.

### 범위

- 오래된 사용자 임시 파일 선택 정리
- 휴지통 용량 확인 및 명시적 비우기
- DNS 캐시 초기화
- 작업 전 미리보기
- 작업 실행 결과 표시
- 관련 진단 재실행
- 작업 전후 결과 비교

### 완료 기준

- 사용자의 명시적 확인 없이 시스템을 변경하지 않는다.
- 허용된 경로 밖의 파일을 삭제하지 않는다.
- 성공, 일부 성공, 실패와 취소를 구분한다.
- 관리 작업 이후 관련 진단 결과의 변화를 확인할 수 있다.

## Phase 3. 공개 품질 안정화

### 목표

개인 개발 환경이 아닌 다른 Windows PC에서도 사용할 수 있는 품질을 확보한다.

### 작업

- 실제 PC 환경별 회귀 테스트
- 오류 처리 및 사용자 메시지 개선
- 접근성 점검
- 개인정보가 포함될 수 있는 로그 검토
- README와 사용 설명서 보완
- 이슈 및 버그 리포트 템플릿 추가
- 오픈소스 기여 가이드 검토
- 첫 공개 버전 릴리스 노트 작성

### 완료 기준

- 알려진 주요 오류와 제한사항이 문서화된다.
- 새로운 사용자가 문서만으로 프로그램을 실행할 수 있다.
- 공개 가능한 첫 버전의 기능과 테스트가 안정화된다.

## Phase 4. CI 및 릴리스 파이프라인 고도화

### 목표

코드 변경마다 빌드와 테스트가 자동으로 검증되도록 한다.

### 작업

- Release 구성 자동 빌드
- 빌드 결과물 보관
- 버전 태그 검증
- 코드 커버리지 및 테스트 결과 게시
- 배포 단계 승인 정책
- 워크플로 권한과 비밀값 관리

### 완료 기준

- Pull Request마다 빌드와 테스트가 자동 실행된다.
- 실패한 빌드가 릴리스 단계로 진행되지 않는다.
- 로컬 환경과 CI 환경의 빌드 결과가 일치한다.

## Phase 5. MSIX 패키징

### 목표

Windows 애플리케이션에 적합한 설치·제거·버전 관리 체계를 구축한다.

### 작업

- MSIX 패키지 프로젝트 구성
- 패키지 식별자와 버전 정책 정의
- 앱 아이콘과 메타데이터 구성
- 설치 및 제거 검증
- 업데이트 시 사용자 설정 유지 검증
- 패키지 서명 방식 검토
- CI에서 MSIX 패키지 생성

### 완료 기준

- 개발 환경이 아닌 PC에서 MSIX로 설치하고 제거할 수 있다.
- 버전이 변경된 패키지로 정상 업데이트할 수 있다.
- CI가 재현 가능한 MSIX 결과물을 생성한다.

## Phase 6. 클라우드 베타 배포

### 목표

정식 출시 전에 사용할 수 있는 독립적인 테스트·베타 배포 채널을 구축한다.

### 후보 구성

- 클라우드 오브젝트 스토리지 또는 정적 배포 공간
- 개발, 베타, 안정 채널 분리
- 버전별 MSIX 패키지 저장
- 업데이트 메타데이터 배포
- 배포 파일 무결성 검증
- 이전 안정 버전 보관
- 배포 실패 시 롤백 절차

### 작업

- Git 태그 또는 릴리스 기준 자동 배포
- CI에서 클라우드 저장소로 패키지 업로드
- 베타 설치 및 업데이트 흐름 검증
- 비밀값과 배포 권한 관리
- 이전 버전으로의 복구 절차 문서화

### 완료 기준

- 지정한 버전이 자동으로 베타 채널에 배포된다.
- 테스트 사용자가 별도 개발 도구 없이 설치할 수 있다.
- 잘못된 배포를 중단하거나 이전 버전으로 복구할 수 있다.

### 미확정 사항

다음 항목은 이 단계에 진입할 때 비교 후 결정한다.

- 사용할 클라우드 제공자
- 패키지 저장 위치
- 업데이트 메타데이터 형식
- 채널 분리 방식
- 서명 인증서 관리 방식

## Phase 7. Microsoft Store 출시

### 목표

일반 Windows 사용자가 신뢰 가능한 경로로 프로그램을 설치하고 업데이트할 수 있도록 한다.

### 작업

- 스토어 등록 정보 준비
- 앱 설명, 아이콘 및 스크린샷 준비
- 개인정보 처리 원칙 공개
- 패키지 제출 및 인증 대응
- 정식 버전 출시
- 업데이트 제출 절차 정리
- 스토어 배포와 베타 채널의 버전 정책 정리

### 완료 기준

- Microsoft Store에서 애플리케이션을 검색하고 설치할 수 있다.
- 스토어를 통한 업데이트가 정상 동작한다.
- 제출 및 인증 과정에서 발견된 문제를 문서화한다.

> Microsoft Store 출시와 WinGet 등록은 정식 릴리스가 준비된 이후
> 진행 상황에 따라 순서를 바꾸거나 병행할 수 있다.

## Phase 8. WinGet 등록

### 목표

명령줄을 선호하는 사용자와 개발자가 WinGet으로 프로그램을 설치할 수 있도록 한다.

### 작업

- 패키지 식별자와 매니페스트 작성
- 설치 및 제거 명령 검증
- 버전별 매니페스트 관리
- 공개 패키지 등록 절차 진행
- 새 버전 릴리스 시 매니페스트 갱신 절차 정리

### 완료 기준

다음 형태의 명령으로 프로그램을 설치할 수 있다.

```powershell
winget install <PackageId>
```

- 설치, 업데이트 및 제거가 정상적으로 동작한다.
- 정식 릴리스와 WinGet 버전 정보가 일치한다.

## Phase 9. 릴리스 자동화 고도화

### 목표

빌드부터 각 배포 채널까지의 릴리스 과정을 안전하게 자동화한다.

### 목표 흐름

```text
Pull Request
→ 자동 빌드 및 테스트
→ main 병합
→ 버전 태그
→ MSIX 생성
→ 베타 채널 배포
→ 검증
→ 정식 릴리스 승인
→ Microsoft Store 및 WinGet 배포
```

### 작업

- 버전 생성 및 검증 자동화
- 릴리스 노트 생성 지원
- 개발·베타·정식 채널 분리
- 수동 승인 단계 적용
- 배포 실패 알림
- 배포 결과 기록
- 롤백 절차 자동화 검토
- 코드 서명 비밀값의 안전한 관리

### 완료 기준

- 동일한 소스 버전에서 재현 가능한 배포 결과물을 만든다.
- 테스트를 통과하지 않은 버전은 배포되지 않는다.
- 정식 배포 전 사람의 승인 단계를 둘 수 있다.
- 배포 이력과 실패 원인을 추적할 수 있다.

## 향후 기능 후보

다음 항목은 MVP와 초기 배포 완료 이후 별도로 검토한다.

- SFC 및 DISM 실행 지원
- 시작 프로그램 비활성화 및 복원
- 관리자 권한 Helper 프로세스
- 배터리 및 전원 상태 진단
- 이전 진단 결과와 변화 비교
- 다크 모드
- 다국어 지원
- 선택적 오류 보고
- 안전한 앱 내 업데이트 확인

기능 후보는 필요성과 안전성이 검증되기 전까지 현재 범위에 포함하지 않는다.

## 장기 완료 모습

WinHealth Inspector는 다음 조건을 갖춘 공개 Windows 애플리케이션을 목표로 한다.

- 실제 사용 가능한 Windows 진단 및 관리 기능
- 테스트 가능한 WPF 애플리케이션 구조
- GitHub Actions 기반 자동 빌드와 테스트
- MSIX 설치 패키지
- 클라우드 기반 베타 배포 채널
- Microsoft Store 정식 배포
- WinGet 패키지 등록
- 버전 관리, 배포 승인 및 롤백 절차
- 공개 소스 코드와 사용자 문서
